# `PriorityQueue` и задачи Top-K

## Цель

Показать, как использовать `PriorityQueue` для задач "топ N", планирования и потоковой обработки, где нужно быстро доставать минимальный/максимальный элемент.

Базовая идея:

- min-heap по умолчанию (`PriorityQueue` без компаратора);
- max-heap через `Comparator.reverseOrder()` или свой компаратор;
- типичный паттерн Top-K: хранить кучу размера `k`.

## Суть подхода

- Куча хранит только элементы, которые потенциально нужны в ответе.
- Для Top-K не держим все `n` элементов: ограничиваемся структурой размера `k`.
- После обработки каждого нового элемента поддерживаем инвариант (размер кучи и правило приоритета).
- Такой подход особенно полезен для потоковых данных, когда весь набор нельзя или не хочется хранить в памяти.

## Примеры

<div class="lenar-tip-grid" markdown="block">

!!! note "Top-K через кучу фиксированного размера"

    Подход поддерживает размер `k`: лишний элемент сразу удаляется.
    Сложность: на каждый элемент `offer/poll` по `O(log k)`, суммарно `O(n log k)`, память `O(k)`.

    ```java
    import java.util.Comparator;
    import java.util.PriorityQueue;

    int[] nums = {5, 1, 9, 3, 7};
    int k = 3;

    PriorityQueue<Integer> heap = new PriorityQueue<>(Comparator.reverseOrder());
    for (int x : nums) {
        heap.offer(x);
        if (heap.size() > k) {
            heap.poll();
        }
    }
    ```

</div>

## Частые ошибки

- Путать порядок итерации `PriorityQueue` с порядком приоритета (итератор не сортирует).
- Выбирать `PriorityQueue` там, где достаточно `ArrayDeque` (обычная FIFO-очередь).
- Хранить слишком большие объекты в куче без необходимости, когда можно хранить ключи/индексы.
