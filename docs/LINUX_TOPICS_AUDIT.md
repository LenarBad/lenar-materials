# Linux Topics Audit

Документ фиксирует findings и backlog по усилению справочного слоя в `docs/linux/topics/*`.

## Область
- Проверяем только `docs/linux/topics/*`.
- SQL topics в этот этап не включены.

## Текущее состояние (критические наблюдения)
- Контент в Linux topics структурно аккуратный, но по глубине это скорее quick reference, чем полноценный canonical reference.
- По большинству тем не хватает справочных "развилок решения": что выбирать в похожих сценариях и почему.
- Мало lookup-формата (таблицы "задача -> команда", "симптом -> проверка -> действие").
- Edge-cases и безопасные практики описаны слишком кратко (rollback, dry-run, минимизация риска, distro-specific нюансы).
- Runbook-последовательности (рабочие цепочки команд под инцидент) представлены слабо.

## Приоритеты доработок
1. **P0 (сначала):** `network`, `troubleshooting`, `security`  
   Причина: максимальная практическая ценность в инцидентах и продакшене.
2. **P1:** `processes`, `files`
3. **P2:** `basics` (уже лучше остальных, но можно усилить)

## Статус реализации
- `P0` - выполнено: `network`, `troubleshooting`, `security` усилены decision guidance, lookup-таблицами и runbook-блоками.
- `P1` - выполнено: `processes`, `files` усилены decision guidance, lookup-таблицами и runbook-блоками.
- `P2` - выполнено: `basics` дополнен shell safety; FHS и раскладка каталогов объединены с темой `files`.

Текущий фокус после `P2`: поддерживать качество как continuous improvement (обновления по мере новых инцидентов и практик).

## Backlog по темам
### Linux Network
- Добавить диагностический flow по уровням: host -> DNS -> route -> port -> local listener -> firewall.
- Добавить матрицу "симптом -> проверка -> следующая команда".
- Расширить примеры с `curl -v`, `traceroute`/`tracepath`, `ip rule`, `resolvectl`.

### Linux Troubleshooting
- Добавить стандартный incident-runbook (фиксировать симптом, baseline, гипотеза, проверка, фиксация результата).
- Добавить таблицу источников диагностики: `journalctl`, `dmesg`, `ss`, `top`, `df/free`, сервисные логи.
- Добавить паттерны "последнее изменение" (конфиг, пакет, релиз, окружение).

### Linux Security
- Добавить baseline hardening checklist (SSH, sudo policy, updates cadence, open ports, лог-аудит).
- Добавить минимальный раздел про `ufw`/`nft`, `fail2ban`, ключи SSH.
- Добавить короткие decision notes (когда парольный вход допустим/недопустим, когда root-login запрещать).

### Linux Processes
- Добавить карту сигналов (TERM/INT/KILL/HUP) и рекомендуемый порядок остановки.
- Добавить блок `systemctl` vs `kill` vs `pkill`.
- Добавить наблюдение ресурсов (`ps` поля, `top/htop`, `nice/renice`).

### Linux Files (включая FHS / раскладку каталогов)
- Добавить блок про symlink/hardlink и когда что использовать.
- Добавить `du`/`df`/`stat`/`xargs` паттерны для операционного использования.
- Добавить безопасные сценарии массовых операций (preview -> apply).

### Linux Basics
- Добавить минимум по shell safety: quoting, wildcard expansion, redirects, exit codes.
- Добавить "быстрый pre-flight" перед изменениями в системе.

## Definition of Done для topic-страниц
Тема считается "усиленной", если:
- есть не только команды, но и decision guidance (что и когда выбирать);
- есть минимум 1 lookup-таблица или эквивалентный компактный справочный блок;
- есть минимум 1 практический runbook-паттерн;
- раздел остается справочным (без возврата к учебным блокам "Когда использовать"/"Пошаговый подход"/"Проверка результата");
- сохранены `Связанные материалы` и `Где использовать`.
