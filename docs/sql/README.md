# SQL

SQL (Structured Query Language) — декларативный язык для работы с реляционными данными: выборки, изменения схемы, контроль доступа и транзакции. В продакшене и аналитике от него зависит и корректность отчётов, и устойчивость нагрузки на СУБД.

Материалы раздела выстроены так же, как в Linux: полные справочные объяснения лежат в **`topics/`**; **`cheatsheets/`** даёт короткие шаблоны; **`recipes/`** — готовые сценарии задач. Учебный трек — в **`learning/`** (развивается отдельно).

## Сильные стороны SQL как инженерного инструмента
- один язык для ad-hoc анализа и для встраивания в приложения;
- явная модель «таблицы — связи — ограничения», удобная для командной работы;
- переносимые идеи (джойны, агрегации, индексы) между разными СУБД с поправкой на диалект.

## Где SQL используется чаще всего
- хранение и выборка данных в веб‑ и backend‑сервисах;
- отчёты, дашборды, продуктовая и операционная аналитика;
- интеграции, ETL/ELT, репликация и контроль качества данных;
- сопровождение инцидентов: «что записалось», «почему дубликаты», «почему медленно».

## Уровень материалов
Ориентир — **середина и выше**: вы уже пишете запросы к реальным таблицам, понимаете джойны и `GROUP BY`, и вам нужны устойчивые шаблоны, типовые ошибки и связь с индексами и планом выполнения. Глубина **не** доведена до уровня внутренностей конкретной СУБД или тонкой настройки кластера.

Примеры в `topics/` стоят **рядом с темой**, которую иллюстрируют; в `cheatsheets/` — компактные заготовки без повторения теории; в `recipes/` — цельные сценарии под задачу. Блок **«основы»** разбит на [несколько узких тем](topics/basics/README.md), чтобы не смешивать структуру запроса, `NULL`, типы и CTE в одной странице.

Где уместно, в темах добавляются **схемы и SVG** (например, диаграмма множеств на странице [Joins](topics/joins/README.md)) — без тяжёлых зависимостей, файлы лежат в `docs/assets/`.
