# SQL Indexes

## Цель
Понимать, зачем в БД индексы, как **составной** индекс обслуживает типичные фильтры и сортировку, и в каких случаях оптимизатор вынужден читать таблицу целиком. Это уровень «инженер + прикладной SQL», без тонкой настройки железа и внутренностей движка.

## Зачем нужен индекс
Индекс — вспомогательная структура (чаще всего B-дерево), которая ускоряет **поиск по ключу** и иногда **упорядочивание** без полного сканирования таблицы. Плата: дополнительное место на диске и стоимость обновления при `INSERT`/`UPDATE`/`DELETE`.

## Первичный ключ и уникальные индексы
- **PRIMARY KEY** обеспечивает уникальность и задаёт основной идентификатор строки.
    - В одних СУБД PK физически определяет размещение данных (кластеризация).
    - В других PK остаётся главным ключом ссылочной целостности и доступа к строке, но без обязательного физического порядка хранения.
- **UNIQUE** запрещает дубликаты по набору столбцов; `NULL` часто обрабатывается особым образом (в некоторых СУБД допускается несколько `NULL` — смотрите документацию).

## Составной индекс и порядок столбцов
Индекс `(a, b, c)` полезен для условий вида `a = …`, `a = … AND b = …` и т.д. слева направо. Запрос только с `b = …` **может** не использовать такой индекс эффективно — ключевое слово «левый префикс».

Практическое правило: первым в индексе ставьте столбец с **высокой селективностью** и тем, что чаще фигурирует в `WHERE`, если нет жёсткого противоречия с паттерном запросов.

**Контраст:** индекс `(user_id, created_at)` хорошо стыкуется с фильтром по пользователю и дате; запрос **только** по `created_at` этот индекс использует иначе (часто нужен отдельный индекс по времени или другой план):

```sql
SELECT id, created_at
FROM orders
WHERE created_at >= TIMESTAMP '2025-04-01'
ORDER BY created_at
LIMIT 500;
```

## Покрывающий индекс (index-only)
Если все нужные столбцы присутствуют в индексе, СУБД может обойтись без обращения к основной таблице (если версия строки видима в транзакции — нюанс MVCC в PostgreSQL и аналоги). На практике это заметно на горячих чтениях.

## Когда индекс часто не помогает
- Условие с **функцией над столбцом**: `WHERE lower(email) = …` — сравнение не по самому столбцу, если нет выраженного/функционального индекса.
- Очень **низкая селективность** (например, флаг с двумя значениями на огромной таблице): планировщик может выбрать последовательное чтение.
- Неподходящий тип сравнения или неявное приведение типов — см. [Типы, даты и литералы](../types-and-literals/README.md).

## Пример: запрос и ожидаемый тип индекса

<div class="lenar-tip-grid" markdown="block">

!!! note "Фильтр по равенству и диапазону, сортировка"

    Индекс вида `(user_id, created_at DESC)` часто соответствует и условию, и `ORDER BY` для одного пользователя — если кардинальность и статистика позволяют планировщику его выбрать.

    ```sql
    SELECT id, created_at
    FROM orders
    WHERE user_id = 12345
      AND created_at >= TIMESTAMP '2025-04-01'
    ORDER BY created_at DESC
    LIMIT 50;
    ```

!!! note "Частичный индекс (PostgreSQL)"

    Имеет смысл, когда **и запросы, и индекс** ограничены одним и тем же предикатом — меньше строк в индексе и быстрее обновление.

    ```sql
    CREATE INDEX orders_open_due_idx
    ON orders (due_at)
    WHERE status = 'open';
    ```

    В других СУБД — свои названия для «условных» индексов.

</div>

## Частые ошибки
- «Навесить индекс на всё подряд» — замедлит запись и усложнит обслуживание; индексы должны соответствовать **реальным** запросам.
- Ожидание, что `OR` по разным столбцам всегда хорошо использует один составной индекс — часто сложнее; иногда нужны отдельные индексы или переписывание условия (зависит от СУБД).
- Игнорирование дублирования строк из джойнов до агрегации — индекс не спасёт от логической ошибки в запросе ([Joins](../joins/README.md)).
