# «NULL», логика и уникальность строк

## Цель
Уверенно писать условия и читать результаты там, где в данных есть **отсутствующее значение** (`NULL`), и понимать, как **`DISTINCT`** определяет «уникальность» строки в результате.

## Почему `NULL` — не «пустая строка» и не ноль
В SQL `NULL` означает «значение неизвестно или не применимо». Это не строка из нуля символов и не число ноль: сравнения с обычными литералами ведут себя иначе.

## Трёхзначная логика в условиях
Для предиката возможны три исхода: **истина**, **ложь**, **неизвестно** (*unknown*). Последний чаще всего возникает, когда в выражении участвует `NULL` (например, `x = NULL` или `x > NULL`).

В **`WHERE`** остаются только строки, для которых предикат **истинен**. Если результат **неизвестен**, строка **отбрасывается** — так же, как при лжи. Поэтому «не равно» через `<> NULL` **не отберёт** строки с `NULL` в поле: сравнение даёт неизвестно, не истину.

## Как проверять «нет значения»
Используйте только:

- `столбец IS NULL`
- `столбец IS NOT NULL`

Не полагайтесь на `= NULL` или `!= NULL`.

## `COALESCE`
`COALESCE(a, b, c, …)` смотрит аргументы **слева направо** и возвращает **первый**, который не `NULL`. Если все `NULL`, результат тоже `NULL`.

Типичные применения: показать пользователю «человекочитаемое» имя из нескольких полей; в расчёте заменить отсутствующую сумму на ноль, чтобы не ломать арифметику; подставить дату по умолчанию для сортировки (осторожно с бизнес-смыслом).

<div class="lenar-tip-grid" markdown="block">

!!! note "Несколько запасных вариантов для подписи"

    Сначала публичное имя, если пусто — логин, если и он не задан — фиксированная строка.

    ```sql
    SELECT id,
           COALESCE(display_name, login, '— без имени') AS label
    FROM users;
    ```

!!! note "Число для вычислений: «пусто» считаем как ноль"

    `discount` может быть `NULL` (скидка не задана); при расчёте итога удобно трактовать это как 0.

    ```sql
    SELECT order_id,
           amount,
           COALESCE(discount, 0) AS discount_for_calc,
           amount - COALESCE(discount, 0) AS to_pay
    FROM orders;
    ```

## `NULL` и агрегаты
Поведение зависит от функции: `COUNT(*)` считает строки группы, а `COUNT(столбец)` — только строки, где в столбце **не** `NULL`. Среднее (`AVG`) по умолчанию не включает `NULL` в знаменатель как отдельные нули. Подробнее — [Aggregation](../aggregation/README.md).

## `SELECT DISTINCT`
`DISTINCT` убирает **полностью совпадающие** строки результата **по всем столбцам**, перечисленным в `SELECT`. Если выбрано `user_id` и `city`, то пара `(1, Moscow)` и `(1, SPb)` — **две разные строки**, обе останутся. Частая ошибка — ждать «уникальность только по `user_id`», оставив в списке лишние столбцы; тогда нужна либо другая проекция, либо группировка/подзапрос.

<div class="lenar-tip-grid" markdown="block">

!!! note "Отбор только заполненных полей и подстановка для отображения"

    ```sql
    SELECT id, email
    FROM users
    WHERE email IS NOT NULL;
    ```

    ```sql
    SELECT id, COALESCE(display_name, login) AS title
    FROM users;
    ```

    Больше вариантов и числовой пример — в блоке **`COALESCE`** выше.

## Частые ошибки
- Фильтрация «не из списка» через `NOT IN (подзапрос)`, который может вернуть `NULL` — итог может стать неинтуитивным; в таких случаях часто яснее **`NOT EXISTS`** ([Joins](../joins/README.md)).
- Ожидание, что `WHERE col != 'x'` вернёт строки с `col IS NULL` — не вернёт из-за трёхзначной логики.
- Ожидание уникальности «по одному полю» при `DISTINCT` с несколькими столбцами в `SELECT`.
