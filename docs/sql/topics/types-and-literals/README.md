# Типы данных, литералы и даты

## Цель
Меньше сюрпризов от **сравнений**, **диапазонов** и **индексов**: согласованные типы в выражениях, предсказуемые границы по времени и осознанное использование `BETWEEN`, литералов и приведения `CAST`.

## Какие типы обычно встречаются
Точные имена зависят от СУБД, но семейства повторяются:

| Семейство | Зачем | Пример смысла |
| --- | --- | --- |
| Строковые | Текст, коды, email | `VARCHAR(n)`, `TEXT` |
| Числовые | Суммы, количества, идентификаторы | `INTEGER`, `BIGINT`, `NUMERIC(p,s)` |
| Логические | Да/нет (если тип есть в диалекте) | `BOOLEAN` |
| Дата/время | Календарная дата, момент времени, интервал | `DATE`, `TIMESTAMP`, `TIMESTAMPTZ`, `INTERVAL` |

Для **денег** предпочтительны типы с фиксированной десятичной точностью (`NUMERIC` / `DECIMAL`), а не «голый» `FLOAT`, чтобы не ловить ошибки округления при расчётах.

## Литералы
**Литерал** — константа прямо в тексте запроса.

- **Строка:** в одинарных кавычках: `'hello'`. Кавычка внутри строки — по правилам диалекта (удвоение `''` или экранирование).
- **Число:** `42`, `-1`, `3.14`; для денег в литерале тоже можно, но тип столбца должен это переварить.
- **Логика:** в PostgreSQL, например, `TRUE` / `FALSE`; в других СУБД иногда `1`/`0` или отдельный тип.
- **Дата/время (стандарт SQL):** `DATE '2025-04-12'`, `TIMESTAMP '2025-04-12 14:30:00'` — удобно, когда формат фиксирован стандартом.
- **Интервал** (если поддерживается): в PostgreSQL часто `INTERVAL '7 days'`, в других СУБД — свой синтаксис.

Тип результата выражения (`столбец + литерал`, функция и т.д.) задаёт правилами СУБД; при сомнении смотрите документацию или план.

<div class="lenar-tip-grid" markdown="block">

!!! note "Литералы разных видов в одном запросе"

    Строка, число и стандартные литералы даты/времени в условии:

    ```sql
    SELECT *
    FROM orders
    WHERE status = 'paid'
      AND amount > 0
      AND created_at >= DATE '2025-01-01'
      AND created_at <  TIMESTAMP '2026-01-01 00:00:00';
    ```

</div>

## Неявное приведение типов
Смешивание **строки и числа** в одном сравнении (например, столбец `VARCHAR` с числом `user_id = 123` или наоборот) заставляет СУБД **угадать**, что к чему привести. Итог зависит от диалекта: индекс по столбцу может не использоваться, сравнение строк с числами даст неожиданный порядок.

**Практика:** приводите к **одному** типу явно и согласуйте тип столбца в схеме с тем, что реально хранится (идентификатор — число, а не «строка из цифр», если нет веской причины).

<div class="lenar-tip-grid" markdown="block">

!!! note "Числовой идентификатор — числовой литерал"

    Если столбец числовой, литерал без кавычек; так меньше шансов на неявное приведение.

    ```sql
    SELECT * FROM orders WHERE user_id = 12345;
    ```

    Если столбец по ошибке строковый, а вы сравниваете с числом, СУБД может привести типы не так, как вы ждёте — лучше поправить схему; временно: `WHERE user_id = '12345'` только если столбец действительно строка.

</div>

## Даты и время
- **`DATE`** — календарный день без времени суток (в типичной семантике).
- **`TIMESTAMP` / `DATETIME`** — момент «на шкале времени» без указания зоны (интерпретация зависит от настроек сеанса).
- **`TIMESTAMPTZ`** (PostgreSQL) и аналоги — момент с учётом часового пояса; хорошо для событий «когда это было в мире».

Хранить дату события в **строке** `VARCHAR` и сортировать как текст — типичный источник багов: строка `'2025-2-1'` лексикографически может идти не так, как хронология. Правильный путь — тип `DATE`/`TIMESTAMP`; пока строка остаётся, фильтровать через явное приведение (см. карточку ниже).

<div class="lenar-tip-grid" markdown="block">

!!! note "Окно «последние 7 суток» от текущего момента (PostgreSQL)"

    Литерал интервала и функции времени; в других СУБД — свой синтаксис для «сейчас» и интервала.

    ```sql
    SELECT *
    FROM events
    WHERE event_ts >= NOW() - INTERVAL '7 days'
      AND event_ts <  NOW();
    ```

!!! note "Дата в строке: временный обход до миграции схемы"

    Если `legacy_date` — `VARCHAR`, запрос может «работать», но сортировка и индексы страдают. Цель — перевести колонку в `DATE`/`TIMESTAMP`; до этого — `CAST` при фильтрации:

    ```sql
    SELECT *
    FROM old_table
    WHERE CAST(legacy_date AS DATE) BETWEEN DATE '2025-01-01' AND DATE '2025-01-31';
    ```

</div>

## `BETWEEN`
`expr BETWEEN a AND b` эквивалентно `expr >= a AND expr <= b` для обычных упорядочиваемых типов — **оба конца включены**. Для `DATE` это «включая первый и последний день»; для `TIMESTAMP` убедитесь, что верхняя граница не отсекает последнюю долю дня, если нужен «весь последний день» — иногда проще полуоткрытый интервал.

<div class="lenar-tip-grid" markdown="block">

!!! note "`BETWEEN` по полю типа `DATE`"

    Весь апрель 2025:

    ```sql
    SELECT *
    FROM orders
    WHERE order_date BETWEEN DATE '2025-04-01' AND DATE '2025-04-30';
    ```

</div>

## Полуоткрытые интервалы
Условие `>= начало AND < конец` не включает правую границу. Удобно для отчётов «за год», «за сутки», чтобы не воевать с `23:59:59.999`.

<div class="lenar-tip-grid" markdown="block">

!!! note "Год целиком по `TIMESTAMP`"

    Все события с 1 января 2025 включительно до (не включая) 1 января 2026:

    ```sql
    SELECT *
    FROM events
    WHERE event_ts >= TIMESTAMP '2025-01-01 00:00:00'
      AND event_ts <  TIMESTAMP '2026-01-01 00:00:00';
    ```

</div>

## Явное приведение: `CAST` и аналоги
`CAST(expr AS тип)` по стандарту SQL. В PostgreSQL часто пишут `expr::тип` — то же по смыслу, короче в интерактиве.

<div class="lenar-tip-grid" markdown="block">

!!! note "Диапазон после приведения к числу"

    Отбор по числовому коду, если в столбце тип «строка, но это числа»:

    ```sql
    SELECT *
    FROM events
    WHERE CAST(event_code AS INTEGER) BETWEEN 100 AND 199;
    ```

!!! note "Момент времени → день для группировки или фильтра"

    ```sql
    SELECT order_id, CAST(created_at AS DATE) AS order_day
    FROM orders
    WHERE created_at >= TIMESTAMP '2025-01-01 00:00:00';
    ```

    В PostgreSQL день из метки часто пишут как `created_at::date`.

</div>

## Связь с индексами
Предикат `WHERE lower(email) = 'x'` использует **значение функции**, а не сырой столбец — обычный B-tree по `email` может не подойти без индекса по выражению. Подробнее — [Indexes](../indexes/README.md).

## Частые ошибки
- Сравнивать дату в строке с датой в типе `DATE` без единого формата и без `CAST`.
- Использовать `BETWEEN` по `TIMESTAMP` для «целого месяца», забыв, что последний день месяца заканчивается в `23:59:59…` — часть строк может выпасть; полуоткрытый интервал или явная верхняя граница «начало следующего месяца».
- Полагаться на неявное приведение в джойне двух таблиц (`ON a.id = b.code`), когда типы разные — сначала выровнять типы.
