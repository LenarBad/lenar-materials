# Linux Files

## Цель
Понимать типичную иерархию каталогов (FHS) — куда смотреть за конфигами, логами и данными сервисов. Уметь выбирать команды для копирования, поиска, прав и массовых операций без лишнего риска.

## Раскладка каталогов (FHS)

### Куда смотреть в первую очередь
| Задача | Где искать | Комментарий |
| --- | --- | --- |
| Конфиг сервиса | `/etc/<service>/`, `/etc/<service>.conf` | локальные правки держите в `/etc`, не в каталогах пакета |
| Логи в файлах | `/var/log/` | плюс `journalctl` для юнитов systemd |
| Разбор падения сервиса | `/var/log`, `journalctl -u <unit>` | сначала журнал, потом файлы |
| Бинарники из пакетов | `/usr/bin`, `/usr/sbin` | |
| Unit systemd (поставка) | `/lib/systemd/system` или `/usr/lib/systemd/system` | путь зависит от дистрибутива |
| Свой override для unit | `/etc/systemd/system` | не правьте vendor-файл в `/lib` или `/usr/lib` |
| Данные приложения | `/var/lib/<service>` | |
| Состояние на время работы (pid, сокеты) | `/run` | не сводите поиск только к `/var` |
| Временные файлы | `/tmp` (часто очищается), `/var/tmp` (дольше живут) | между перезагрузками — лучше `/var/tmp` |

### Дистрибутивы
- **Debian/Ubuntu и RHEL-family** делят одну FHS-базу; отличается в первую очередь полный путь к unit в `/lib` vs `/usr/lib`.
- **Пакеты** ставятся через разные менеджеры (`apt` / `dnf`), но целевые каталоги конфигов и логов обычно совпадают.

### Остальная карта (кратко)
`/`, `/home`, `/root` — корень и домашние каталоги. `/usr` — программы и библиотеки; `/bin`, `/sbin` часто указывают на `/usr`. `/var` — изменяемые данные (логи, кэш, очереди). `/opt` — стороннее ПО. `/dev`, `/proc`, `/sys` — устройства и интерфейсы к ядру.

### Ориентир по системе и сервису
Сначала общий вид, затем цепочка «конфиг → лог → unit → данные → runtime» для выбранного имени сервиса (`S` замените на своё):

```bash
ls -la / /etc /var/log /home 2>/dev/null | head -n 80
ls -la /usr/bin | head
systemctl cat ssh 2>/dev/null | head -n 20
ls -la /run | head

S=myapp
ls -la /etc/$S 2>/dev/null
ls -la /var/log | rg $S
journalctl -u $S -n 50 --no-pager
systemctl cat $S
ls -la /var/lib/$S 2>/dev/null
ls -la /run | rg $S
```

## Операции: что выбирать
| Сценарий | Команда | Когда |
| --- | --- | --- |
| Копия на том же хосте | `cp -a src dst` | нужны атрибуты и структура |
| Перенос / переименование | `mv src dst` | одна файловая система (одна точка монтирования) |
| Зеркало каталогов | `rsync -a --delete src/ dst/` | повторяемая синхронизация |
| Поиск по условиям | `find ...` | фильтры и массовые действия |
| Объём по ветке | `du -sh *` | найти «тяжёлые» каталоги |
| Метаданные объекта | `stat file` | права, inode, время |

### `rsync -a --delete src/ dst/`
- `-a` — рекурсивно и с атрибутами; `--delete` — в `dst/` убрать лишнее относительно `src/`.
- Слеш в `src/` — копировать **содержимое**, не саму директорию.
- Перед боевым запуском: `rsync -a --delete --dry-run src/ dst/` (тот же принцип предпросмотра, что и для других опасных массовых команд).

## Массовые и разрушающие действия
1. Предпросмотр списка затронутых путей (`--dry-run` у `rsync`, `find … -print` без удаления и т.п.).
2. Сохранить список в файл и просмотреть.
3. Выполнить действие **по этому списку**, не меняя условий между шагами.
4. Проверить итог (`ls`, `stat`, `du`, при необходимости `df`).

Пример для старых логов: [безопасная чистка старых логов](../../recipes/safe-old-logs-cleanup.md).

## Примеры

<div class="lenar-tip-grid" markdown="block">

!!! note "Просмотр, поиск и размер"

    Содержимое текущего каталога: права, владельцы, скрытые файлы.

    ```bash
    ls -lah
    ```

    Конфиги в `/etc`, имя заканчивается на `.conf` (ошибки доступа в сторону скрываем).

    ```bash
    find /etc -name "*.conf" 2>/dev/null
    ```

    Размер каждого элемента в `/var/log`, сортировка по возрастанию — удобно увидеть самые «тяжёлые».

    ```bash
    du -sh /var/log/* | sort -h | tail
    ```

!!! note "Копирование и перенос"

    Копия каталога `app` с атрибутами в `app.bak`.

    ```bash
    cp -a app app.bak
    ```

    Зеркало `app/` в резерв — лишнее в назначении удалится (`--delete`).

    ```bash
    rsync -a --delete app/ /backup/app/
    ```

    Перенос бэкапа в `/tmp` (путь подставьте под свою задачу).

    ```bash
    mv app.bak /tmp/
    ```

!!! note "Ссылки и метаданные"

    Символическая ссылка на конфиг в текущем каталоге.

    ```bash
    ln -s /etc/myapp/config.yml ./config.link
    ```

    Права, inode, время изменения и т.д.

    ```bash
    stat /etc/myapp/config.yml
    ```

!!! note "Права и владелец"

    Текущая маска прав для новых файлов (что «срезается» при создании).

    ```bash
    umask
    ```

    Права на файл: владелец читает/пишет, группа только читает, остальные не видят.

    ```bash
    chmod 640 /etc/myapp.conf
    ```

    Владелец и группа — `root` (для сервисных конфигов типичный вариант).

    ```bash
    chown root:root /etc/myapp.conf
    ```

## Частые ошибки
- класть важное в `/tmp` и терять после очистки или перезагрузки; путать `/tmp` и `/var/tmp`;
- править конфиги не в `/etc` или править unit в каталоге пакета вместо `/etc/systemd/system`;
- путать системные логи в `/var/log` с файлами в `/home`; искать runtime только в `/var`, забывая про `/run`;
- `rm -rf` без финальной проверки пути; массовые операции без списка и предпросмотра;
- `cp -r` там, где нужны атрибуты (`cp -a` / `rsync -a`); `chmod 777` без необходимости; не менять владельца после копирования, если сервис от него зависит.
