# Linux Troubleshooting

## Цель
Дать системный алгоритм диагностики Linux-проблем: быстро сузить область поиска и выйти на первопричину.

## Incident runbook
1. Зафиксировать симптом и время начала проблемы.
2. Собрать baseline: статус сервиса, ресурсы, сеть, последние ошибки.
3. Сформировать 1 гипотезу и проверить ее минимальным изменением.
4. Подтвердить результат и зафиксировать изменение.
5. Если не помогло, откатить и перейти к следующей гипотезе.

## Источники диагностики
| Что проверяем | Команда | Что получаем |
| --- | --- | --- |
| Статус unit | `systemctl status <service>` | текущее состояние и последние ошибки |
| Логи unit | `journalctl -u <service> -n 200 --no-pager` | хронология сбоев сервиса |
| Системные сообщения | `dmesg | tail -n 100` | OOM, I/O, kernel warnings |
| Ресурсы RAM/CPU | `free -h`, `top` | давление по памяти и процессам |
| Диск и inode | `df -h`, `df -i` | переполнение FS/inode |
| Сокеты | `ss -tulpen` | слушающие порты и процессы |

## Симптом -> проверка -> действие
| Симптом | Проверка | Действие |
| --- | --- | --- |
| `service failed` после рестарта | `systemctl status`, `journalctl -u` | исправить конфиг/секреты, повторить запуск |
| Рост latency/timeout | `top`, `ss`, `iostat` (если доступно) | локализовать CPU/RAM/IO/NETWORK bottleneck |
| `No space left on device` | `df -h`, `df -i` | очистить логи/временные файлы, настроить ротацию |
| Периодические 502/504 | `ss`, healthcheck через `curl -v` | проверить backend listener и upstream timeout |
| После деплоя все сломалось | `journalctl --since`, diff конфига | откатить изменение и проверить гипотезу повторно |

!!! note "Baseline сервиса"

    Текущее состояние unit и последние сообщения статуса (`myapp` замените на своё имя).

    ```bash
    systemctl status myapp --no-pager
    ```

    Хронология сбоев сервиса в journal.

    ```bash
    journalctl -u myapp -n 200 --no-pager
    ```

!!! note "Системные сигналы и ресурсы"

    Сообщения ядра (OOM, диск, драйверы) — последние строки.

    ```bash
    dmesg | tail -n 50
    ```

    Память: суммарно и по зонам.

    ```bash
    free -h
    ```

    Заполнение файловых систем.

    ```bash
    df -h
    ```

    Интерактивно: процессы и нагрузка (выход — `q`).

    ```bash
    top
    ```

!!! note "Сетевой слой"

    Кто слушает порты и с какими параметрами (адрес, процесс).

    ```bash
    ss -tulpen
    ```

!!! note "Unit и конфиг"

    Итоговый unit-файл (drop-in, override), как реально видит systemd.

    ```bash
    systemctl cat myapp
    ```

    Свежие файлы в каталоге конфигурации приложения.

    ```bash
    ls -lt /etc/myapp
    ```

## Последнее изменение: быстрый чек

Время начала инцидента — подставьте своё; дальше команды используют переменную в оболочке.

```bash
INCIDENT_SINCE="2026-04-10 10:20:00"
```

Файлы в `/etc`, у которых время модификации позже этой отметки (поиск свежих правок конфигов).

```bash
sudo find /etc -type f -newermt "$INCIDENT_SINCE" -printf "%TY-%Tm-%Td %TH:%TM %p\n" | sort
```

Недавние установки/обновления пакетов (Debian/Ubuntu: `dpkg.log`).

```bash
sudo zgrep -h " upgrade " /var/log/dpkg.log* | tail -n 50
```

Логи unit с момента инцидента — сопоставление с деплоем или рестартом.

```bash
journalctl -u myapp --since "$INCIDENT_SINCE" --no-pager | tail -n 100
```

Текущее состояние DNS и резолвера на хосте.

```bash
resolvectl status
```

Переменные окружения, которые systemd передаёт unit (если заданы).

```bash
systemctl show myapp -p Environment
```

Права на каталог и файлы конфигурации приложения.

```bash
sudo ls -l /etc/myapp /etc/myapp/*
```

Что проверяем:
- Конфиг: появились ли изменения в `/etc` прямо перед инцидентом.
- Пакеты: не было ли обновлений в тот же интервал.
- Деплой: совпадает ли время релиза с ростом ошибок.
- Окружение: не изменились ли DNS, env, секреты или права доступа.

## Частые ошибки
- сразу применять "рандомный фикс" без подтвержденной причины;
- не фиксировать исходное состояние до изменений;
- менять несколько параметров одновременно;
- не делать rollback после неуспешной гипотезы;
- не сохранять минимальный постмортем по итогу инцидента.
