# Linux Processes

## Цель
Понять, как запускать, наблюдать и останавливать процессы, чтобы контролировать нагрузку и стабильность сервисов.

## Что такое unit (systemd)
`Unit` - это объект управления в `systemd`: то, что можно запускать, останавливать, перезапускать и проверять через `systemctl`.

Частые типы unit:

- `service` - сервис/демон (например, `nginx.service`);
- `socket` - сокет-активация;
- `target` - группа зависимостей и состояние системы;
- `timer` - запуск по расписанию;
- `mount` - точка монтирования.

Быстрые команды:

- `systemctl status <name>` - статус unit;
- `systemctl restart <name>` - перезапуск;
- `systemctl enable <name>` - автозапуск при загрузке;
- `systemctl cat <name>` - итоговая конфигурация unit и override.

## Что выбирать в типовых развилках
| Сценарий | Команда | Когда использовать |
| --- | --- | --- |
| Найти процесс по имени | `pgrep -af <name>` | быстрый поиск PID и аргументов |
| Детально посмотреть процесс | `ps -fp <PID>` | владелец, PPID, время старта |
| Остановить сервис корректно | `systemctl stop <service>` | если процесс управляется systemd |
| Завершить одиночный процесс | `kill -TERM <PID>` | мягкое завершение без потери состояния |
| Завершить группу процессов | `pkill -f <pattern>` | точечное завершение нескольких процессов |
| Принудительная остановка | `kill -KILL <PID>` | только если TERM не сработал |

## Карта сигналов
| Сигнал | Назначение | Практика |
| --- | --- | --- |
| `TERM` | корректное завершение | первый выбор |
| `INT` | прерывание (как Ctrl+C) | для foreground-задач |
| `HUP` | перечитать конфиг/перезапуск сессии | daemon reload по документации сервиса |
| `KILL` | немедленное завершение | последний шаг, без graceful shutdown |

## Симптом -> проверка -> действие
| Симптом | Проверка | Действие |
| --- | --- | --- |
| Высокий CPU | `ps -eo pid,ppid,cmd,%cpu,%mem --sort=-%cpu | head` | локализовать процесс и снизить приоритет/перезапустить |
| Высокая память | `ps -eo pid,cmd,%mem,rss --sort=-%mem | head` | проверить утечку, restart с фиксацией логов |
| Сервис "упал" | `systemctl status`, `journalctl -u` | исправить причину и запускать через `systemctl` |
| Процесс не завершается | `kill -TERM`, затем `kill -KILL` | эскалация сигналов с паузой между шагами |
| Много "зомби" | `ps -el | rg ' Z '`, `ps -fp <PPID>` | проверить родительский процесс и supervision |

## Примеры

<div class="lenar-tip-grid" markdown="block">

!!! note "Поиск процесса"

    Список PID и полной командной строки по шаблону имени:

    ```bash
    pgrep -af nginx
    ```

    Детали одного процесса (здесь — первый PID среди совпадений `nginx`):

    ```bash
    ps -fp "$(pgrep -o nginx)"
    ```

!!! note "Нагрузка и «тяжёлые» процессы"

    Интерактивный обзор загрузки (выход — `q`):

    ```bash
    top
    ```

    Снимок: процессы с наибольшим потреблением CPU:

    ```bash
    ps -eo pid,cmd,%cpu,%mem --sort=-%cpu | head
    ```

!!! note "Остановка и эскалация сигналов"

    Сначала мягкое завершение (сигнал `TERM`):

    ```bash
    kill -TERM <PID>
    ```

    Пауза, чтобы процесс успел завершиться:

    ```bash
    sleep 3
    ```

    Если не помогло — принудительное завершение (сигнал `KILL`):

    ```bash
    kill -KILL <PID>
    ```

!!! note "Сервис под systemd"

    Текущее состояние unit:

    ```bash
    systemctl status nginx
    ```

    Перезапуск после правок конфигурации и т.п.:

    ```bash
    sudo systemctl restart nginx
    ```

    Последние строки журнала сервиса:

    ```bash
    journalctl -u nginx -n 100 --no-pager
    ```

</div>

## Частые ошибки
- сразу использовать `kill -9` без попытки нормального завершения;
- останавливать PID, не проверив что это нужный процесс;
- перезапускать сервис без просмотра текущего статуса и логов;
- использовать `kill` для процессов, которыми управляет systemd;
- не фиксировать PID/команду до принудительного завершения.
