# Образы и контейнеры

## Цель
Понять практическую разницу между образом и контейнером, как версионировать образы и как быстро проверить состояние запущенного экземпляра.

## Базовая модель
- **Образ** (`image`):
    - Неизменяемый шаблон: файловая система + метаданные запуска.
    - Обычно содержит: стартовую команду, объявленные порты, переменные окружения, рабочую директорию.
    - Обычно хранится в registry (Docker Hub, GHCR, private registry).

- **Контейнер**:
    - Конкретный экземпляр образа, который запускается как процесс на хосте.
    - Имеет собственное runtime-состояние: процессы, PID, сетевые интерфейсы, статус.
    - Использует writable-слой поверх read-only слоёв образа.
    - Имеет имя/ID и жизненный цикл (`created` → `running` → `exited` → `removed`).

- **Практические последствия**:
    - Из одного образа можно запустить много независимых контейнеров.
    - Изменения внутри контейнера (например, `apt install`) исчезают после удаления контейнера, если не пересобрать образ.
    - Данные приложения сохраняют в томах, а не только во внутренней файловой системе контейнера.

## Теги и воспроизводимость
У образа есть два способа ссылки:

- **тег** (`:1.27-alpine`, `:latest`) — удобное имя версии;
- **digest** (`@sha256:...`) — точный идентификатор содержимого.

Ключевая разница:

- тег можно переназначить на другой образ;
- digest всегда указывает на один и тот же набор слоёв.

Практическое правило:

- локальная разработка: обычно достаточно тега;
- CI/CD и прод: лучше фиксировать digest, чтобы выкатывался ровно проверенный образ.

```bash
# Удобно читать, но тег может "переехать"
docker pull docker.io/library/nginx:1.27-alpine

# Строго фиксированная версия содержимого
docker pull docker.io/library/nginx@sha256:abcdef...
```

Как читать этот пример:

- Команда с тегом (`:1.27-alpine`) удобна для человека, но через время может скачать уже другой образ с тем же тегом.
- Команда с digest гарантирует побитово тот же образ, который вы уже протестировали.

Проверка, что запустили именно ожидаемый образ:

```bash
docker image inspect docker.io/library/nginx:1.27-alpine --format '{{.Id}}'
docker image inspect docker.io/library/nginx@sha256:abcdef... --format '{{.Id}}'
```

Если ID совпали, это один и тот же локальный образ.

## Типичный цикл работы

```bash
# 1) Загружаем образ на хост
docker pull docker.io/library/nginx:1.27-alpine

# 2) Создаём и запускаем контейнер в фоне
docker run -d --name web -p 8080:80 docker.io/library/nginx:1.27-alpine

# 3) Проверяем, что контейнер реально в статусе Up
docker ps

# 4) Смотрим последние логи контейнера
docker logs --tail 50 web

# 5) Останавливаем и удаляем контейнер
docker stop web && docker rm web
```

Что здесь важно:

- `docker run` одновременно делает create + start.
- `-p 8080:80` пробрасывает порт хоста `8080` в порт контейнера `80` (сервис слушает внутри контейнера, а вы заходите с хоста).
- После `docker rm web` образ `nginx:1.27-alpine` остаётся локально (его нужно удалять отдельно: `docker rmi ...`).

## Что именно проверять после запуска

Быстрые команды для диагностики:

```bash
# Статус контейнера, PID процесса и код последнего выхода
docker inspect web --format 'status={{.State.Status}} pid={{.State.Pid}} exit={{.State.ExitCode}}'

# Какие порты реально опубликованы
docker port web

# Какие изменения в writable-слое контейнера
docker diff web
```

Интерпретация:

- `status=running` и `pid` > 0 — основной процесс контейнера жив.
- `docker port web` должен показать соответствие `80/tcp -> 0.0.0.0:8080` (или конкретному IP).
- `docker diff` показывает изменения относительно образа (`A` добавлен файл, `C` изменён, `D` удалён).

## Сценарий: почему контейнер «сразу падает»

Типовая причина: основной процесс внутри контейнера завершился, поэтому контейнер перешёл в `exited`.

Минимальная проверка:

```bash
docker ps -a --filter "name=web"
docker logs web
docker inspect web --format 'status={{.State.Status}} exit={{.State.ExitCode}} error={{.State.Error}}'
```

Если `exit` не `0`, смотрите логи и команду запуска (`docker inspect web --format '{{.Path}} {{join .Args " "}}'`) — часто ошибка в параметрах, переменных окружения или недоступном файле конфигурации.

## Частые ошибки
- Путать образ и контейнер: удаление контейнера не удаляет образ.
- Считать, что изменения в контейнере «автоматически попадут» в образ.
- Ждать стабильности от `latest` в CI/CD.
- Публиковать порт `-p` и ожидать доступности сервиса, когда процесс внутри контейнера уже завершился.
- Сохранять важные данные только внутри контейнера, без томов и резервного копирования.
