# Контейнер не стартует

Контейнер завершается сразу или в цикле перезапуска. Цель — **за минуты** понять: это команда, конфиг, порты, тома или образ.

## Когда использовать
- `docker run` завершился, а вы ожидали, что сервис будет работать непрерывно (статус `Up`).
- `docker compose up` показывает `Restarting` или сервис падает при старте.

## См. тематические источники
- [Сборка и Dockerfile](../topics/build-and-dockerfile/README.md) — `CMD`/`ENTRYPOINT` и запуск процесса.
- [Тома и данные](../topics/volumes-and-data/README.md) — bind-mount и права.
- [Сеть контейнеров](../topics/networking/README.md) — порты и доступность.
- [Шпаргалка по CLI](../cheatsheets/docker-cli.md) — команды для логов и отладочного входа.

## Пошаговый подход

1. **Посмотреть код выхода и логи**
    - `docker ps -a` — статус и имя контейнера.
    - `docker logs <имя>` — сообщения процесса и стек при падении приложения.
    - Код выхода (после остановки):

      ```bash
      docker inspect <имя> --format '{{.State.ExitCode}} {{.State.Error}}'
      ```

    - Последние строки без «прокрутки» всего файла:

      ```bash
      docker logs --tail 200 <имя> 2>&1
      ```

2. **Понять, что вообще запускается**
    - Если вы переопределили команду в `docker run …` или в Compose `command:`, она **заменяет** поведение по умолчанию образа — сравните с [темой сборки](../topics/build-and-dockerfile/README.md) про `CMD`/`ENTRYPOINT`.
    - Для «одноразовых» задач образ мог завершаться **корректно** (например, миграции) — тогда «падения» нет, контейнер просто **exit 0**.
    - Убедиться, какая команда у контейнера на самом деле:

      ```bash
      docker inspect <имя> --format '{{json .Config.Cmd}} {{json .Config.Entrypoint}}'
      ```

3. **Проверить порты и адреса**
    - Ошибка «address already in use» — порт на хосте занят; смените левую часть `-p`.
    - Приложение слушает только `127.0.0.1` внутри контейнера — с хоста через `-p` может быть недоступно; настройте bind на `0.0.0.0` внутри приложения (если это допустимо по политике безопасности).
    - Проверить, что слушает порт **внутри** работающего контейнера:

      ```bash
      docker exec <имя> sh -c 'ss -lntp || netstat -lntp'
      ```

4. **Проверить тома и права**
    - Bind-mount с пустым каталогом на хосте может **перекрыть** файлы из образа — приложение не находит конфиг или статику.
    - Права на файлы при bind-mount — см. [тома и данные](../topics/volumes-and-data/README.md).
    - Быстро сравнить «есть ли файл в контейнере»:

      ```bash
      docker exec <имя> ls -la /app
      ```

5. **Запустить интерактивно с shell**
    - Временно: `docker run --rm -it --entrypoint sh <образ>` (или другой shell из образа), затем руками запустить команду сервиса и посмотреть ошибку.
    - Для Compose — тот же образ и переменные, что у сервиса:

      ```bash
      docker compose run --rm --entrypoint sh api
      ```

6. **Compose: сервис в цикле перезапуска**
    - Логи только этого сервиса: `docker compose logs -f --tail 100 api`.
    - Если падение из-за «БД ещё не готова», типичный симптом — ошибка подключения в первых секундах; лечится retry в приложении, healthcheck + `depends_on` с условием готовности (в зависимости от версии Compose) или отдельным init-шагом.

## Проверка результата
- Контейнер в `docker ps` в статусе `Up`, логи без повторяющегося stack trace, сервис отвечает на ожидаемом порту или сокете.
