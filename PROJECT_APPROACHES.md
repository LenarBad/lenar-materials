# Подходы к реализации проекта справочных материалов

## 1) Назначение проекта
Проект предназначен для создания и практического использования справочных материалов по разным областям знаний.

Первая итерация:
- `linux`
- `sql`

Ключевая идея: материалы лежат в отдельных каталогах по областям, имеют единый формат и легко используются в реальных задачах.

## 2) Цели и критерии качества
- Быстрый доступ к нужной информации "по задаче"
- Единая структура в каждой области (чтобы не переучиваться при переходе между темами)
- Практическая ориентация: примеры команд, типовые сценарии, частые ошибки
- Простота расширения: добавление новых областей без переделки всего проекта

## 3) Варианты структуры проекта

### Вариант A: Плоская структура по темам (простой старт)
Подходит для первых недель, когда важно быстро начать.

Пример:
- `linux/`
  - `README.md`
  - `commands.md`
  - `troubleshooting.md`
- `sql/`
  - `README.md`
  - `queries.md`
  - `performance.md`

Плюсы:
- Минимум организации, легко поддерживать на старте
- Быстрое добавление материалов

Минусы:
- Со временем файлы могут стать слишком большими
- Сложнее масштабировать при росте количества тем

### Вариант B: Модульная структура (рекомендуется)
Подходит для системного роста и долгосрочного использования.

Пример:
- `linux/`
  - `basics/`
  - `files/`
  - `processes/`
  - `network/`
  - `security/`
  - `troubleshooting/`
- `sql/`
  - `basics/`
  - `joins/`
  - `aggregation/`
  - `indexes/`
  - `performance/`
  - `troubleshooting/`

Плюсы:
- Хорошая масштабируемость
- Четкая навигация по подтемам

Минусы:
- Чуть выше порог входа при создании
- Нужно заранее поддерживать единый шаблон

### Вариант C: По типу использования (учеба / шпаргалка / кейсы)
Подходит, если проект используется и для обучения, и для быстрого поиска решений.

Пример внутри каждой области:
- `learning/` — короткая теория
- `cheatsheets/` — быстрые команды/паттерны
- `recipes/` — готовые сценарии "как сделать X"

Плюсы:
- Удобно для разных режимов работы
- Хорошо подходит для практического применения

Минусы:
- Есть риск дублирования между разделами
- Нужна дисциплина в правилах наполнения

### Вариант D: Полное совмещение B + C (целевой)
Используем оба измерения полностью:
- **измерение 1 (B):** тематические модули;
- **измерение 2 (C):** тип использования (`learning`, `cheatsheets`, `recipes`).

Идея: внутри каждой области есть единая точка входа и два понятных маршрута:
- "Мне нужна тема" -> иду по модульной структуре;
- "Мне нужен формат потребления" -> иду по `learning/cheatsheets/recipes`.

Плюсы:
- Подходит для разной аудитории (обучение и быстрый практический доступ)
- Масштабируется при росте количества тем
- Удобна для будущего статического сайта и поиска

Минусы:
- Выше требования к правилам навигации и линковки
- Нужна жесткая дисциплина против дублирования контента

## 4) Рекомендуемый стандарт документа в теме
Для единообразия каждый материал может включать:
- Цель: что решает документ
- Когда использовать
- Пошаговый подход
- Примеры (команды, запросы)
- Частые ошибки
- Проверка результата
- Связанные материалы

## 5) Предлагаемая минимальная стартовая реализация (MVP)
1. Создать каталоги областей:
   - `linux/`
   - `sql/`
2. В каждой области завести:
   - `README.md` (главная карта области)
   - `topics/` (измерение B)
   - `learning/`, `cheatsheets/`, `recipes/` (измерение C)
3. Определить единый шаблон документа и правила перекрестных ссылок.
4. Добавить первые 5-10 практических сценариев на область.
5. Проверить, что одну и ту же информацию не дублируем, а связываем ссылками.

## 6) Подход к развитию контента
- Итерационный: небольшие регулярные обновления вместо редких больших
- Практический приоритет: сначала закрывать наиболее частые задачи
- Единый стиль: одинаковые заголовки и формат примеров
- Периодический рефакторинг: разбивать перегруженные документы

## 7) Вопросы для уточнения перед масштабированием
- Кто основная аудитория (только вы, команда, публичный доступ)?
- Нужны ли версии/уровни (junior/middle/senior)?
- Нужны ли тестовые задания и проверочные чек-листы?
- Нужна ли автоматическая публикация (например, в wiki/сайт)?

## 8) Рекомендация на текущем этапе
Зафиксировать целевой подход: **Вариант D (полное совмещение B + C)**.

Правило наполнения:
- `topics/` хранит "источник истины" по теме;
- `learning/`, `cheatsheets/`, `recipes/` дают разные форматы потребления;
- повторяющиеся фрагменты не копируются, а ссылаются на тематический источник.

Это позволяет обслуживать разную аудиторию без потери структуры.

## 9) Целевая структура каталогов (пункт 2)
Ниже базовая структура для первых областей.

```text
linux/
  README.md
  topics/
    basics/
      README.md
    files/
      README.md
    processes/
      README.md
    network/
      README.md
    security/
      README.md
    troubleshooting/
      README.md
  learning/
    linux-fundamentals.md
  cheatsheets/
    core-commands.md
  recipes/
    common-admin-tasks.md

sql/
  README.md
  topics/
    basics/
      README.md
    joins/
      README.md
    aggregation/
      README.md
    indexes/
      README.md
    performance/
      README.md
    troubleshooting/
      README.md
  learning/
    sql-fundamentals.md
  cheatsheets/
    query-patterns.md
  recipes/
    daily-analytics-tasks.md
```

Минимальные правила для этой структуры:
- каждый файл в `learning/cheatsheets/recipes` должен ссылаться на релевантные документы в `topics/`;
- в `topics/*/README.md` должен быть блок "Где использовать" со ссылками на материалы форматов C;
- новые области добавляются по той же схеме без изменения общей архитектуры.
